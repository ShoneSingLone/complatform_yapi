把 `vue-easytable-master` 中 `ve-table` 的能力，按 m2o / xUI 的规范，逐步移植为 `xTableEasy`。

本次计划的目标不是直接宣称“哪些已经实现”，而是先建立一套**可执行、可验证、可回退**的迁移蓝图，后续严格按这个蓝图推进。

## 3. 本次迁移的核心判断

### 3.1 当前最大风险点

当前最大风险点不是 `i18n` 或 emitter，而是：

> **`xTableEasy.vue` 的异步壳转换是否完整。**

需要重点核对：

- `_.$importVue(...)` 依赖列表是否完整；
- 返回解构顺序是否与依赖顺序严格一致；
- 源库中的普通 JS 依赖，是否都已经被正确替换为项目内可用模块；
- JSX 转 `render(h)` 后，是否存在行为偏差；
- 根组件是否完整承接了子模块联动所需的上下文与实例能力。

### 3.2 当前不应再做的事

后续不再采用以下做法：

1. 不再直接相信 `xTableEasy.readme.md` 的“已实现”勾选；
2. 不再仅凭“没看到 mixin”就判定事件链缺失；
3. 不再一上来只修根组件表面报错，而忽略整体模块结构；
4. 不再为了适配 xUI 一次性重组目录结构，导致无法对照源库。

### 3.3 本轮新增确认的结构性风险与修复进展

在继续梳理功能真值表后，本轮新增确认并处理了几项关键问题：

1. **已补齐一批缺失的 `.vue` 包装文件**
    - `selection/constant.vue`
    - `editor/constant.vue`
    - `locale/index.vue`
    - `utils/resize-observer-polyfill.vue`

    这些文件的目的，是让当前 `async function + _.$importVue(...)` 体系下的导入路径，能和项目内真实文件结构对齐。

2. **已通过脚本完成一轮内部 `.vue` 动态导入扫描**
    - 新增脚本：`scripts/check-x-table-easy-imports.js`
    - 当前扫描结果：
        - 扫描到 `.vue` 文件数：199
        - 命中内部 `.vue` 导入数：102
        - 缺失导入数：0

    这说明当前 `xTableEasy` 目录内的内部 `.vue` 动态导入路径，已经没有明显缺文件问题。

3. **当前仍需保留为结构性风险的项**
    - `header-filter-custom-content.vue` 与源实现存在运行时差异：
        - 当前版本没有像源文件一样显式导入 `VeDropdown`、`VeIcon`
        - 而是使用 `h("ve-dropdown")`、`h("ve-icon")` 这种字符串组件名
        - 是否可靠依赖于运行时全局注册，因此必须进入页面验证，不适合仅靠静态代码直接判断通过。
    - `business_doc` 仍存在同一路径重复声明：
        - `routes.vue` 中多个 `x-table-easy` 功能路径同时指向英文聚合页和中文旧示例页
        - 后续如果不先统一正式入口，会影响文档维护、页面回归和自动化测试基线。

4. **编辑器里的 ts-plugin 报错暂按误报处理**
    - 新补的 `.vue` 包装文件在编辑器中出现了 `A module cannot have multiple default exports`
    - 但从现有 `util/constant.vue`
      的同类写法、项目运行模式和脚本扫描结果看，当前更像是 IDE/ts 插件对这种异步壳写法的不兼容误报
    - 在页面运行前，先不把它当成阻塞结论。

这几项说明：

> 当前下一步不应直接跳进“大量功能实现”，而应先做**运行时验证**，确认新补包装文件与关键高风险模块在页面里是否真正可用。

## 4. 总体迁移原则

### 原则 1：以源库为真值

迁移时优先参考：

- `vue-easytable-master/packages/ve-table/src`
- `vue-easytable-master/examples/src/docs/zh/ve-table`
- 当前 xTableEasy 目录下的实际实现

### 原则 2：先建立功能真值表，再动代码

先把功能梳理清楚，再逐项确认：

- 源库有没有；
- 当前目录里有没有对应模块；
- 当前是完整、半成品还是不可用；
- 下一步是补齐、重写还是保留。

### 原则 3：尽量沿用原文件结构

迁移时尽量保持源库模块边界，例如：

- 根入口仍然对应 `ve-table/src/index.jsx`；
- `header/`、`body/`、`footer/`、`selection/`、`column-resizer/`、`util/`、`utils/` 分模块对应迁移；
- 工具模块优先保持原职责，不做无关重组。

### 原则 4：页面实测高于静态猜测

即使静态代码看起来没问题，也必须以页面行为为最终判断依据。

### 原则 5：计划文档可持续修订

如果你的 README 或开发说明存在歧义，我会：

1. 直接在计划文档或你的文档基础上做修订建议；
2. 标出我理解后的版本；
3. 及时向你确认；
4. 由你判断我理解得是否正确。
