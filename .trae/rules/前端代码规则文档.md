## 2. 模块加载与依赖管理

-   **核心方法**：`_.$importVue()` - 异步加载 Vue 组件和模块

-   **路径前缀**：使用 `@/` 前缀表示项目根目录

### 2.2 依赖来源

-   **内部依赖**：`@/` 路径下的项目内部模块

### 2.3 全局对象

-   **`_`**：全局工具对象，提供核心方法

## 3. 组件系统

### 3.1 组件注册

-   **动态注册**：使用 `Vue.component(name, () => _.$importVue(componentURL))`
-   **懒加载**：组件按需加载，提高首屏性能
-   **组件命名**：采用 PascalCase 命名规范

### 3.2 状态管理机制

-   **核心机制**：使用 Vue 原生的 `provide/inject` API 进行组件间状态传递
-   **实现方式**：
    -   父组件通过 `provide()` 方法提供状态
    -   子组件通过 `inject` 选项接收状态
    -   支持跨多层组件传递状态
-   **全局状态**：顶层组件通常会 provide `APP` 实例，子组件通过 inject 接收
-   **组件实例传递**：父组件可以将自身实例 provide 给子组件，实现组件间通信

### 3.3 provide/inject 使用示例

**父组件提供状态：**

```vue
<script lang="ts">
export default async function () {
	return {
		provide() {
			return {
				inject_project: this
				// 其他需要传递的状态或方法
			};
		},
		data() {
			return {
				// 组件状态
			};
		},
		methods: {
			// 组件方法
		}
	};
}
</script>
```

**子组件接收状态：**

```vue
<script lang="ts">
export default async function () {
	return {
		inject: ["APP", "inject_project"],
		methods: {
			someMethod() {
				// 使用注入的状态和方法
				console.log(this.inject_project.someData);
				this.inject_project.someMethod();
			}
		}
	};
}
</script>
```

### 3.3 组件结构

-   **单文件组件**：标准 `.vue` 文件格式
-   **TypeScript 支持**：使用 `<script lang="ts">` 标签（仅为 IDE 识别，不经过构建处理）
-   **script 部分通用写法**：所有 .vue 文件的 script 部分使用以下通用结构：
    ```javascript
    export default async function ({ PRIVATE_GLOBAL }) {
    	// 组件逻辑
    	return defineComponent({
    		// 组件选项
    	});
    }
    ```
    -   **参数**：`PRIVATE_GLOBAL` 是可选参数，根据业务需求使用
    -   **返回值**：根据业务需求返回组件定义或其他值
-   **script 部分限制**：
    -   只能使用浏览器原生支持的 JavaScript 语法
    -   无法使用需要编译的 TypeScript 特性（如泛型、命名空间、装饰器）
    -   只能使用 JSDoc 这种浏览器原生的注解标准
    -   无法使用 ES6+ 特性中浏览器不支持的部分
-   **样式管理**：`<style lang="less">`，支持嵌套和变量

## 6. 状态管理

### 6.1 状态管理方式

-   **核心机制**：使用 Vue 原生的 `provide/inject` API 进行组件间状态传递
-   **组件状态**：使用组件内部 data 和 computed 管理局部状态
-   **全局状态**：通过顶层组件 provide `APP` 实例，子组件 inject 接收
-   **应用状态**：在根组件中管理用户信息、菜单状态等全局数据

### 6.2 provide/inject 最佳实践

-   **命名规范**：注入的状态使用 `inject_` 前缀命名，如 `inject_project`
-   **传递方式**：父组件提供自身实例，子组件通过注入访问父组件的状态和方法
-   **使用场景**：适合跨多层组件传递状态，避免 props 层层传递的问题
-   **限制**：状态变更追踪困难，不利于调试，大型应用中可能导致状态管理混乱

### 6.3 响应式数据

-   **数据定义**：在组件的 data() 方法中定义
-   **计算属性**：使用 computed 处理派生状态
-   **监听机制**：使用 watch 监听数据变化

## 10. 构建与部署

### 10.1 构建流程

-   **无 package.json**：不使用标准 npm 构建流程
-   **seed.js 加载**：通过 `/common/libs/seed.js` 动态加载应用
-   **版本控制**：HTML 中包含构建版本信息

### 10.2 .vue 文件执行机制

-   **执行方式**：`_.$importVue()` 会调用 .vue 文件导出的异步函数
-   **参数传递**：`_.$importVue()` 会传递包含 `PRIVATE_GLOBAL` 的对象作为参数
-   **执行流程**：
    1. `_.$importVue()` 加载 .vue 文件
    2. 调用 `export default async function ({ PRIVATE_GLOBAL })`
    3. 执行内部的模块加载和初始化逻辑
    4. 接收返回的组件定义或其他值
    5. 根据返回值进行相应处理（如注册组件）
-   **返回值**：根据业务需求返回组件定义或其他值，无强制要求

## 11. 与标准前端项目的差异

### 11.1 核心差异

| 特性              | business_xspace                                      | 标准前端项目                                 |
| ----------------- | ---------------------------------------------------- | -------------------------------------------- |
| 模块加载          | `_.$importVue()` 动态加载                            | ES6 import 或 require                        |
| 依赖管理          | 无 package.json，使用公共库                          | package.json + npm/yarn/pnpm                 |
| 构建工具          | 无构建配置，动态加载                                 | webpack/vite/rollup                          |
| 入口文件          | entry.vue + seed.js                                  | main.js + index.html                         |
| .vue 文件导出结构 | `export default async function ({ PRIVATE_GLOBAL })` | `export default {}` 或 `defineComponent({})` |
| 组件注册          | 运行时动态注册                                       | 编译时静态分析                               |
| TypeScript 支持   | 仅 IDE 识别，不经过构建处理                          | 完整的 TypeScript 编译流程                   |

### 11.2 优势与劣势

**优势：**

-   灵活性高，无需构建过程
-   模块按需加载，首屏性能好
-   共享公共依赖，减少重复代码
-   动态版本管理，便于缓存控制

**劣势：**

-   缺少类型检查和构建时优化
-   依赖管理不清晰，版本控制困难
-   开发工具支持有限
-   无法使用现代前端生态工具链

## 12. 开发建议

### 12.1 代码组织

-   遵循现有目录结构和命名规范
-   使用 `@/` 路径前缀引用项目文件
-   保持组件职责单一，避免过大的组件

### 12.2 TypeScript 使用建议

-   **语法限制**：只能使用浏览器原生支持的 JavaScript 语法
-   **避免使用**：
    -   高级 TypeScript 特性（如泛型、命名空间、装饰器）
    -   ES6+ 特性中浏览器不支持的部分
    -   需要编译处理的语法糖
-   **注解标准**：只能使用 JSDoc 这种浏览器原生的注解标准
-   **IDE 支持**：利用 IDE 的 TypeScript 提示功能，但确保代码可在浏览器中直接运行
-   vue 文件里面 template 里面不可以使用`反引号
-   当 style 或 class 需要动态,使用计算属性 computed 或者方法 method

### 12.3 模块加载

-   使用 `_.$importVue(url)` 加载组件和模块
-   url 以 `/common` 或 `@/` 开头
    -   `/common` - 表示从公共库路径加载（如 `/common/ui-x/xButton.vue`）
    -   `@/` - 表示从项目根目录加载（如 `@/components/MyComponent.vue`）
-   优先使用异步加载，提高应用性能
-   注意路径大小写，保持一致性

### 12.4 API 调用

-   统一使用 `_api.xspace` 调用后端接口
-   实现适当的错误处理和加载状态
-   复用现有的上传和请求逻辑

### 12.5 性能优化

-   合理使用组件懒加载
-   优化图片和资源加载
-   避免不必要的全局状态

**标准对话框组件示例**：

```vue
<template>
	<xDialog style="--xDialog-wrapper-width: 800px">
		<!-- 使用 xForm 组件 -->
		<xForm col="1">
			<!-- 使用 xItem 组件 -->
			<xItem :configs="form.title" />
			<xItem :configs="form.content" />
		</xForm>

		<!-- 对话框底部按钮 -->
		<template #footer>
			<!-- 使用 xBtn 组件 -->
			<xBtn :configs="btnOk" />
			<xBtn @click="closeModal">{{ i18n("cancel") }}</xBtn>
		</template>
	</xDialog>
</template>

<script lang="ts">
export default async function ({ PRIVATE_GLOBAL, closeModal, documentId, onSave }) {
	/* 必要，混入"closeModal", "$layerMax", "$layerMin", "$layerRestore" */
	const { useDialogProps } = await _.$importVue("/common/utils/hooks.vue");

	return defineComponent({
		inject: ["APP"],
		props: useDialogProps(),
		data() {
			const vm = this;
			return {
				form: defItems({
					title: {
						label: i18n("文档标题"),
						value: "",
						rules: [_rules.required()]
					},
					content: {
						label: i18n("文档内容"),
						type: "textarea",
						value: "",
						rules: [_rules.required()]
					}
				}),
				btnOk: {
					label: i18n("ok"),
					preset: "blue",
					async onClick() {
						const [atLestOne] = await _.$validateForm(vm.$el);
						if (atLestOne) return;

						const formData = _.$pickFormValues(vm.form);
						if (onSave) {
							onSave(formData);
						}
						closeModal();
					}
				}
			};
		}
	});
}
</script>
```

## 14. 总结

business\*xspace 前端项目采用了一种独特的动态加载架构，通过 `_.$importVue()` 方法实现模块的按需加载，避免了传统前端项目
的构建过程，提高了开发和部署的灵活性。同时，项目保持了清晰的目录结构和代码组织，便于维护和扩展。

在进行 DocFlow 功能移植时，需要充分理解这种架构特点，适配现有的模块加载机制，优先复用 ui-x 组件库中的组件，同时保持代码
风格和开发规范的一致性。
