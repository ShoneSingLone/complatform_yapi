# Desktop Workspace 对比分析 Spec

## Why

分析参考项目 (`src 的副本`) 与当前业务项目 (`static/business_xspace`) 的桌面工作空间实现差异，找出差距并制定改进方案。

## What Changes

-   **架构差异**：参考项目使用 Vue 3 + Composition API + Pinia，当前项目使用 Vue 2.7 + Options API + 自定义状态管理
-   **图标库**：参考项目使用 lucide-vue-next，当前项目使用内联 SVG
-   **样式系统**：参考项目使用 BEM 命名规范 + CSS 变量，当前项目使用 Less 嵌套
-   **窗口拖拽**：参考项目使用 @vueuse/core 的 useDraggable，当前项目使用原生拖拽实现
-   **组件结构**：参考项目更模块化，当前项目功能更完整（多桌面、背景设置等）

## Impact

-   **需要迁移的技术**：Pinia 状态管理、@vueuse/core 工具库、lucide-vue-next 图标库
-   **需要保留的功能**：多桌面支持、桌面背景自定义、键盘快捷键、状态持久化
-   **需要改进的代码**：窗口拖拽、响应式系统、代码组织结构

## 对比分析

### 1. 技术栈对比

| 特性     | 参考项目        | 当前项目                 | 差距                                    |
| -------- | --------------- | ------------------------ | --------------------------------------- |
| Vue 版本 | Vue 3           | Vue 2.7.16               | **重大**：无法直接迁移 Composition API  |
| 状态管理 | Pinia           | 自定义 Store             | **中等**：Pinia 更现代，但当前方案可用  |
| 图标库   | lucide-vue-next | 内联 SVG                 | **小**：内联 SVG 无依赖，但维护成本高   |
| 工具库   | @vueuse/core    | 原生实现                 | **中等**：@vueuse 提供更好的工具函数    |
| 样式规范 | BEM + CSS 变量  | Less 嵌套                | **小**：当前项目支持 Less，样式系统完整 |
| 模块加载 | Vite + ESM      | \_.$importVue() 动态加载 | **重大**：架构差异，无法改变            |

### 3. 代码质量对比

**参考项目优势**：

-   使用 TypeScript 类型系统完整
-   Composition API 代码更简洁
-   BEM 命名规范清晰
-   样式与结构分离（通过 import）
-   使用现代工具库（@vueuse、lucide）

**当前项目优势**：

-   功能更完整（多桌面、背景、快捷键）
-   状态持久化到服务器
-   符合项目特定的动态加载架构
-   更好的响应式支持

### 4. 具体实现差异

#### Dock 组件

-   **参考项目**：使用 TransitionGroup 和动态组件，样式通过 import 引入
-   **当前项目**：使用 transition-group 和内联 SVG，样式内联
-   **差距**：参考项目代码更简洁，但功能相当

#### Window 组件

-   **参考项目**：使用 useDraggable 实现拖拽，代码简洁
-   **当前项目**：原生拖拽实现，支持更多功能（大小调整）
-   **差距**：当前项目功能更强，但代码复杂

#### DesktopIcon 组件

-   **参考项目**：使用动态组件渲染图标
-   **当前项目**：使用内联 SVG
-   **差距**：参考项目更灵活，当前项目无外部依赖

#### Store

-   **参考项目**：Pinia store，类型安全
-   **当前项目**：普通 JS 对象，响应式通过 watch 实现
-   **差距**：**主要差距所在**，当前项目响应式不够优雅

## ADDED Requirements

### Requirement: 改进状态管理

**优先级：P0**

当前项目的状态管理使用普通 JS 对象，导致需要 watch 和 $forceUpdate 来更新 UI。

**改进方案**：

1. 使用 Vue 2 的响应式 API（Vue.set、this.$set）
2. 或者引入轻量级状态管理库（如 vue-observable）
3. 优化数据结构，确保响应式更新

### Requirement: 优化窗口拖拽

**优先级：P1**

当前项目使用原生拖拽，代码复杂但可以工作。

**改进方案**：

1. 保持当前实现（功能完整）
2. 代码重构，提取公共逻辑
3. 添加更多测试

### Requirement: 统一样式规范

**优先级：P2**

当前项目样式系统完整，但可以借鉴参考项目的 BEM 命名。

**改进方案**：

1. 保持 Less 嵌套语法
2. 采用 BEM 命名规范
3. 提取公共样式变量

## MODIFIED Requirements

### Requirement: 图标系统

**当前实现**：内联 SVG **建议修改**：保持内联 SVG，但统一图标命名和使用方式

**原因**：

-   项目架构限制，无法引入外部图标库
-   内联 SVG 无依赖，性能更好
-   保持当前实现，优化代码组织

## REMOVED Requirements

### Requirement: 迁移到 Composition API

**原因**：项目使用 Vue 2.7.16，不支持 Composition API 的完整特性 **迁移**：保持 Options API，使用最佳实践

### Requirement: 使用 Pinia

**原因**：项目架构限制，无法引入 Pinia **迁移**：优化当前状态管理，使用 Vue 响应式 API

## 总结

### 当前项目优势

1. ✅ 功能更完整（多桌面、背景、快捷键、持久化）
2. ✅ 符合项目架构（动态加载、无构建）
3. ✅ 响应式支持更好

### 需要改进的方面

1. ❌ 状态管理不够优雅（需要 watch + $forceUpdate）
2. ❌ 代码组织可以更清晰
3. ❌ 缺少类型系统支持

### 建议

**不建议完全照搬参考项目**，因为：

1. 技术栈不同（Vue 2 vs Vue 3）
2. 架构不同（动态加载 vs 标准构建）
3. 当前项目功能更完整

**应该**：

1. 保持当前架构和功能
2. 优化状态管理，使用 Vue 响应式 API
3. 改进代码组织和命名规范
4. 保持内联 SVG 和 Less 样式系统
